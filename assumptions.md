# Assumptions

While studying the CodeJudge dataset and designing the relational database schema, several assumptions were made to ensure proper normalization, consistency, and maintainability of the database.
Since the dataset mainly provides raw CSV data, some design decisions were inferred based on common database design practices and the relationships observed between tables.
---

## Student Related Assumptions

- Each student is assumed to have a unique student ID generated by the system.
- Roll numbers are considered unique for every student within the institution.
- Student email addresses are assumed to be unique and are used for communication and identification purposes.
- A student can belong to only one academic batch at a time.
---

## Batch and Course Assumptions

- Each batch represents a specific academic program and admission year.
- A course can be associated with multiple students through enrollments.
- Credit values for courses are assumed to be positive integers.
---

## Enrollment Assumptions
- A student should not enroll in the same course more than once.
- Enrollment records are assumed to track only valid students and valid courses.
---

## Problem and Contest Assumptions

- Each coding problem is identified using a unique problem ID and problem code.
- Problem difficulty levels are assumed to contain only:
  - Easy
  - Medium
  - Hard
- Contest start time must always be earlier than contest end time.
---

## Submission Assumptions
- Every submission belongs to one student.
- while practice submissions score should not be added.
- Submission scores are assumed to be non-negative values.
- A student is allowed to make multiple submissions for the same problem, best one will be taken in the account.
- Programming language values are assumed to contain only valid supported languages.
---

## Test Case and Result Assumptions

- Each coding problem can contain multiple test cases.
- Test results are generated separately for each testcase execution.
- Test results are linked to both submissions and test cases.
---

## Attendance and Session Assumptions

- Attendance is recorded once per student for every session when entirely attended thorughout the session.
- Attendance status values are assumed to contain only:
  - Present
  - Absent
  - Late
- Every session belongs to a valid course.
---

## Regrade and Plagiarism Assumptions

- Regrade requests are associated with individual submissions only.
- Plagiarism flags are generated per submission.
- Similarity percentages are assumed to range between 0 and 100.
- Flagged submissions are assumed to be reviewed manually by administrators or instructors.
---

## Data Import Assumptions

- The `raw_student_import` table is treated as a temporary staging table used during data import operations.
- This table may remain partially denormalized because it stores raw imported data before validation and cleaning.
- Original CSV files are assumed to remain unchanged during the normalization process.

---

## General Database Assumptions

- Foreign key constraints are used to maintain referential integrity between related tables.
- NULL values are allowed only for optional attributes.
- The final schema is designed approximately up to Third Normal Form (3NF) while still keeping the structure practical and easy to maintain.
